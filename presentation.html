05 Key Learnings

Architecture

Container != Worker

Worker code deploys instantly. Container keeps running with OLD env vars until explicitly restarted. Rotating a secret requires container restart, not just redeploy.

File Paths

Two "Workspace" Dirs

/root/clawd/ = user workspace (R2 synced)

/root/.openclaw/workspace/ = OpenClaw's internal workspace where identity files (IDENTITY.md, SOUL.md) must live

Config

Env Vars are King

The startup patcher overwrites provider, channel, and auth config on every boot. Manual openclaw.json edits to those fields won't survive a restart. Use secrets.

Validation

Add New Providers to Checks

The worker validates required env vars on every request. Adding MiniMax required updating validateRequiredEnv() or the "Configuration Required" page blocks everything.

Docker

BuildKit Required

Wrangler uses --load flag which needs Docker BuildKit (buildx). Legacy builder fails. Also need docker group membership in WSL.

CF Access

AUD Must Match Exactly

Deleting/recreating the container app can generate a new Durable Object namespace, which may invalidate the CF Access AUD tag. Always re-verify after container changes.

09 Always-On: Cron Keep-Alive

The Problem

Discovery

Container Sleeps Overnight

  • Slack uses Socket Mode (WebSocket inside container)
  • No incoming HTTP requests at night → DO evicted
  • WebSocket drops → Slack messages lost
  • Heartbeats and cron jobs stop running
  • keepAlive: true isn't enough alone
Solution

5-Minute Cron Trigger

  • Pings Worker every 5 min → keeps DO alive
  • If gateway is down, starts it automatically
  • Slack WebSocket stays connected 24/7
  • Cron triggers are free (count as requests)

What We Added

// wrangler.jsonc - cron trigger "triggers": { "crons": ["*/5 * * * *"] } // src/index.ts - scheduled handler async scheduled(event, env, ctx) { // Touch the sandbox DO to keep it alive const sandbox = getSandbox(env.Sandbox); const proc = await findExistingProcess(sandbox); if (!proc || proc.status !== 'running') { // Gateway died, restart it ctx.waitUntil(ensureGateway(sandbox, env)); } }

Cost Impact

ComponentCost
Cron requests (288/day)Free
Container memory (4 GiB 24/7)~$26/mo
CPU (idle ~2-5%)~$1/mo
Disk (8 GB)~$1.50/mo
Always-on total~$28-29/mo
vs. sleep mode (4h/day)~$5-6/mo
Why not webhooks? Slack Socket Mode maintains a persistent WebSocket from inside the container — there's no webhook URL the Worker can receive. The container must stay alive to receive messages. The cron is the simplest, cheapest way to guarantee that.

11 Fix: Cron & Heartbeat Never Fired

Bug

Symptoms

  • Cron jobs never ran automatically
  • Heartbeat never sent a Slack message
  • Manual cron triggers worked fine
  • Slack responses worked fine
Root Cause

OpenClaw 2026.2.3 Scheduler Bugs

  • Silent-death race — scheduler crashed on startup with no error (fixed in 2026.2.13)
  • Stall on error — one failed run froze the scheduler permanently (fixed in 2026.2.14)
  • Heartbeat & cron are separate from the gateway — both rely on the same internal scheduler
Fix

One Line in Dockerfile

# Before RUN npm install -g openclaw@2026.2.3 # After RUN npm install -g openclaw@2026.2.17

Rebuilt and redeployed. Cron and heartbeat both started firing immediately.

Heartbeats ≠ Cron triggers. They are independent features, both driven by OpenClaw's internal scheduler (croner). Slack working was misleading — the gateway handles incoming messages without the scheduler. Keep OpenClaw up to date; openclaw update exists but won't persist in a container — bump the Dockerfile version instead.

12 Fix: Cloudflare Browser Rendering

Problem

Browser Tool Always Failed

  • Built-in browser tool needs local Chrome + extension relay — not available in container
  • CF Access blocked the /cdp WebSocket endpoint (302 redirect)
  • R2 was restoring old skill scripts on every boot, overriding image updates
  • WORKER_URL secret had token baked in, breaking WebSocket URL construction
Architecture

How CDP Works

Container script (screenshot.js) ↓ wss:// + CF-Access-Client-Id/Secret headers Worker /cdp endpoint (WebSocket) ↓ Puppeteer API Cloudflare Browser Rendering
Fixes

What We Changed

  • CF Access service token — created cdp-client token, added Bypass policy to main Access app
  • Skill scripts updated — send CF-Access-Client-Id / CF-Access-Client-Secret headers on WebSocket connect
  • Bundled skills staging — skills copied to /opt/openclaw-bundled-skills/ in image, applied after R2 restore so updates are never overwritten
  • /cdp HTTP response — returns CDP discovery JSON (not error) so OpenClaw reachability check passes
  • SOUL.md — explicit instruction to use skill scripts, never the built-in browser tool
CF Access path bypass doesn't work for Workers apps. Path rules are only supported for SaaS/Tunnel apps. The correct pattern is a service token bypass policy on the main application — machine-to-machine auth via headers, no OAuth required.
